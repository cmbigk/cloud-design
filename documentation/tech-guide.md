# 📘 Technical Operations Guide

This guide provides the necessary commands and information to manage your cloud-design microservices project running on AWS.

## 1. Overview of the Architecture
- **Infrastructure:** Provisioned via Terraform on a `t3.small` Spot Instance.
- **Microservices (6 total):**
    - `api-gateway` (Port 3000 -> Exists on Port 80 via Ingress)
    - `inventory-app` (Port 8080)
    - `billing-app` (Port 8080)
    - `inventory-db` (Port 5432)
    - `billing-db` (Port 5432)
    - `rabbit-queue` (Port 5672)
- **Container Registry:** Private AWS ECR repositories.
- **Networking:** K3s with Traefik Ingress.

## 2. Managing the Cluster Locally
To interact with Kubernetes from your local machine, use the `kubeconfig` generated by the `scripts/deploy.sh` script.

```bash
# Set your Kubeconfig environment variable
export KUBECONFIG=$(pwd)/kubeconfig/k3s.yaml

# Check cluster nodes
kubectl get nodes

# See all running microservices
kubectl get pods

# See external access points
kubectl get ingress
```

## 4. Troubleshooting & Logging
If a service behaves unexpectedly, you can view the logs:
```bash
# View Gateway logs
kubectl logs -l app=api-gateway

# View Billing Worker logs
kubectl logs -l app=billing-app
```

## 5. Development Workflow (How to Update)
1. **Change Code:** Edit files in `src/`.
2. **Rebuild & Push:** Run `./scripts/push_images.sh` to update ECR.
3. **Redeploy:** 
   ```bash
   kubectl rollout restart deployment api-gateway inventory-app billing-app
   ```

## 6. Cleanup (Avoiding Costs)
To completely delete the AWS infrastructure once you are finished with the project:
```bash
cd terraform
terraform destroy -auto-approve
```
## 7. Audit Defense (Potential Questions)
Here are suggested answers for the questions in `task/audit.md`:

- **Why AWS?** AWS offers the most mature "Free Tier" and credit system, and the Elastic Container Registry (ECR) integrates perfectly with our Docker workflow.
- **How did you manage costs?** By using **Spot Instances** for the EC2 node, we reduced compute costs by 70%. We also used K3s instead of EKS to avoid the $72/month EKS control plane fee.
- **How is it secure?** All databases and message queues are isolated inside the VPC. They are NOT reachable from the internet—only the API Gateway can talk to them.
- **Why Kubernetes (K3s)?** K3s provides a standard Kubernetes API while being lightweight enough to run on a single small instance, making it perfect for cost-effective microservice orchestration.
- **What about Scalability?** Since we use Kubernetes manifests, we can scale pods using `kubectl scale`. If we need more nodes, we could expand the Terraform code to an Auto-Scaling Group.
